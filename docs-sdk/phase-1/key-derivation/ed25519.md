# Ed25519 Key Derivation

**Phase:** 1 - Cryptographic Fundamentals  
**Status:** ✅ Done (`0.0.4-dev`)  
**Implementation:** [`lib/src/crypto/xrpl_ed25519.dart`](https://github.com/nemorixgroup/xrpl-flutter-sdk/blob/main/lib/src/crypto/xrpl_ed25519.dart)

## What This Is

XRPL's newer signing algorithm, the default for many client libraries
(`xrpl.js` among them). Derivation is much simpler than secp256k1's:
one hash produces the secret key directly, with no root/intermediate
split and no validity-check retry loop, because (per the official
spec) every 32-byte number is technically a valid Ed25519 secret key.

## The Algorithm

```
seed entropy (16 bytes)
v SHA-512Half
32-byte secret key  <- used directly, no validation/retry needed
v standard Ed25519 (RFC 8032) public key derivation
32-byte public key
v prefix with 0xED
33-byte public key (matches secp256k1's compressed key length)
```

## Decision Taken

```dart
static Future<XrplEd25519KeyPair> deriveKeyPair(XrplEntropy entropy) async {
  final secretKeyBytes = XrplHash.sha512Half(entropy.bytes);
  final keyPair = await _algorithm.newKeyPairFromSeed(secretKeyBytes);
  final extractedPublicKey = await keyPair.extractPublicKey();
  return XrplEd25519KeyPair(
    privateKey: secretKeyBytes,
    publicKey: Uint8List.fromList(extractedPublicKey.bytes),
  );
}
```

`package:cryptography`'s `Ed25519().newKeyPairFromSeed()` treats our
already-computed `secretKeyBytes` as a standard RFC 8032 seed and runs
the standard Ed25519 key generation on it - this is exactly the
"standard public key derivation for Ed25519" the official XRPL spec
calls for.

## Why This Method Is `async` (and secp256k1's Are Not)

`package:cryptography` does not support secp256k1 at
all, so this couldn't be made symmetric with `XrplSecp256k1` by
switching libraries. The alternative that *is* synchronous
(`ed25519_edwards`) has had no real updates in about four years, which
we judged too risky for cryptographic code. `XrplWallet` (`0.0.5-dev`)
will hide this difference behind one consistently async public API.

## The `0xED` Prefix

`XrplEd25519KeyPair.prefixedPublicKey` prepends `0xED` to the raw
32-byte public key, producing 33 bytes - matching secp256k1's
33-byte compressed public key length. This is required by the
official spec so that downstream code (address derivation in Phase 2,
for example) can handle both algorithms' public keys as a uniform
33-byte value without a special case for Ed25519.

## Test Vectors / Verification

Checked against a key pair computed independently via Python's
`pynacl` (a binding to libsodium, a widely audited reference Ed25519
implementation), using the same official Ed25519 seed entropy already
verified in
[family-seed-encoding](../family-seed-encoding/README.md) (from the
official seed `sEdTM1uX8pu2do5XvTnutH6HsouMaM2`, taken directly from
the `ripple-address-codec` test suite):

- Secret key (`SHA-512Half` of the entropy)
- Raw 32-byte public key
- Full 33-byte `0xED`-prefixed public key

5 tests total. See
[`test/crypto/xrpl_ed25519_test.dart`](https://github.com/nemorixgroup/xrpl-flutter-sdk/blob/main/test/crypto/xrpl_ed25519_test.dart).
